### 1.1 [Mostly Automatic Translation of Language Interpreters from C to Safe Rust](https://arxiv.org/abs/2606.27122) — Wang, Paulsen, Dodds, Kroening, Mathur, arXiv:2606.27122, 2026

**Authors.** Bo Wang, Brian Paulsen, Joshua Dodds, Daniel Kroening, Umang Mathur. Kroening (Amazon Web Services / formerly Oxford, CBMC/ESBMC lineage) is the followed name; the rest are AWS Automated Reasoning collaborators plus academic co-authors — a strong verification-flavoured byline for a C-to-Rust migration paper.

**Venue.** arXiv:2606.27122 (cs.PL), submitted 20 June 2026. No conference target announced in the abstract; the closest fit given the empirical + verification angle is PLDI, OOPSLA, or FSE 2027.

**License.** arXiv non-exclusive distribution licence — figures may not be embedded verbatim; any structural depiction below is a Mermaid recreation.

**Problem.** C-to-Rust migration has become one of the most active applied-PL topics of the last two years (see C2Rust, Laertes, VERT, CRust, and the recent LLM-driven Vert/Rustify variants), but every existing pipeline hits the same wall: **safe** Rust requires satisfying the borrow checker, and real C programs — especially interpreter cores — encode ownership and aliasing patterns that no purely syntactic translator can lift. Interpreters matter because they (i) handle untrusted user input by design and (ii) concentrate the memory-unsafety risk of an entire language ecosystem in a single binary; a mujs / lua-interp / duktape vulnerability is a wide-blast-radius incident. The prior LLM-based translators were either evaluated on toy sizes or accepted `unsafe` blocks in the output, defeating the point.

**Approach — feature-reduction milestones plus multi-agent validate-and-refine.**

Reboot's core design idea is to reject the "translate the whole file in one shot" framing that every previous LLM-driven translator adopts. Instead:

1. **Feature reduction.** Decompose the C program into a sequence of feature-restricted subsets: a version that only exercises arithmetic, then one that adds control flow, then one that adds heap allocation, then one that adds pointer aliasing, etc. Each subset is a *complete, testable* program. Translation proceeds from the smallest subset upward; every milestone must pass the (subsetted) test suite before the next feature is restored. This bounds the search space at every step and gives the multi-agent loop a hard, executable acceptance criterion.
2. **Multi-agent orchestration.** A driver agent dispatches translation subtasks to worker coding agents, then routes their output through automated validators (Rust compiler + test suite + memory-safety sanitiser). Failures come back as structured feedback the driver can attribute to a specific milestone. Because milestones are small and validators are deterministic, the loop terminates in bounded time even though the coding agents themselves are unreliable.

**Headline numbers.** Six real-world interpreters translated: `mujs`, plus five others in the 6k–23k LoC range. Each translation required **1 to 11 brief user interventions** (the abstract's phrase). All six pass **100%** of the provided test suites; on **separately created validation tests never exposed to the system** they pass **62%–92%**. The security case study on `mujs` shows the C-side memory vulnerabilities (heap buffer overflow, use-after-free) are **eliminated** in the safe-Rust output — the borrow checker + Rust std collections rule them out by construction.

**Why this is Outstanding for this reader.** Three reasons converge. First, the *acceptance criterion* is deterministic (compilation, tests, sanitiser) — this is the same "make the LLM's output validatable by a non-LLM tool" pattern the pipeline flagged as the deciding structural move in the 2026-06-30 Revelio report; two of the highest-impact papers this quarter share that architectural instinct. Second, interpreters are a *load-bearing* migration target — a working translation of `mujs` or `duktape` immediately becomes a candidate replacement in real embedded and browser-adjacent settings. Third, the reported 62–92% validation-set pass rate is honest — most prior LLM-translator papers report on the training tests only, and 62% on unseen tests is a much more useful number.

**Open questions.** The abstract does not disclose (i) which Rust toolchain features and lifetime patterns the system prefers (raw pointers via `NonNull`? `Rc<RefCell<...>>`? arenas?), (ii) whether the "brief user interventions" scale sub-linearly with LoC — 11 interventions on 23k lines is remarkable, but the same rate on 200k lines is 100+ interventions, and (iii) how well the pipeline handles C interpreters that JIT-compile (which requires `unsafe` for W-to-X memory) — none of the six interpreters listed in the abstract look like JIT-heavy targets, so this may be an out-of-scope condition. The full PDF is the natural next read.

**Verdict.** Outstanding — the strongest applied-PL result from the followed-researcher feed this month, with a deterministic acceptance criterion that generalises beyond translation, and a security case study that connects the PL work to a memory-safety incident category we care about.

### 2.1 [Axon: A Synthesizing Superoptimizer for Tensor Programs](https://arxiv.org/abs/2606.26344) — Kothari, Zhu, Kroening, Sung, arXiv:2606.26344, 2026

**Authors.** A. Kothari, S. Zhu, D. Kroening, C. Sung. Kroening on the byline again — this is the second Kroening-affiliated arXiv drop in the same 24 h window, both from the AWS Automated Reasoning line.

**Venue.** arXiv:2606.26344, submitted June 2026. Natural conference targets: PLDI, ASPLOS, MLSys 2027.

**License.** arXiv non-exclusive — no figure embedding.

**Problem.** Writing high-performance kernels for AI accelerators (TPU-class, tile-based DSAs, IREE targets) requires deep expertise in tiling, instruction selection, data layout, and operator fusion. Existing autotuning stacks (Ansor, TVM AutoTVM, Tenset, TC) rely on hand-crafted rewrite rules to explore semantically equivalent variants; new hardware means new rules. Superoptimisation for scalar ISA (Souper, STOKE, alive2) does not lift to tensor programs because the rewrite domain is unbounded (arbitrary shape and stride).

**Approach.** Axon combines three ideas:

- *Algebraic transformation discovery via propagation.* Operators are propagated through the computation graph, and semantic equivalence between variants is checked by **SMT over unbounded tensors** — no hand-crafted rewrite rules required. This is the paper's main soundness contribution: the equivalence checker is total over the (parametric) input shape space, not a bounded set of examples.
- *Target ISA lowering.* Tensor operations are lowered to target-specific instructions with tiling configurations constrained by a hardware descriptor. The paper implies (but does not detail in the abstract) that the descriptor covers tile sizes, register file, and DMA characteristics.
- *Fusion.* Operators and instructions are fused to minimise memory traffic — the standard latency-hiding move on tile-based accelerators, but here integrated into the same synthesis loop as the equivalence check.

**Why Keep.** For a reader whose main axis is PL / program synthesis / verification, the interesting object is the *SMT-over-unbounded-tensors* soundness argument — this is a genuine technical contribution and, if the encoding is reusable, will show up in downstream systems. The abstract stops before revealing the empirical numbers, so headline speedups and coverage are not yet quotable; the full PDF is worth the follow-up read.

**Open questions.** (i) What's the theory used for unbounded tensors — parametric arrays over integer indices? A dedicated tensor-logic SMT decision procedure? (ii) How does the equivalence checker time-out gracefully on shape-dependent transformations that cross an undecidable boundary? (iii) What's the target hardware — the abstract says "tile-based AI accelerators" without naming a chip.

### 2.3 [To Run or Not to Run: Analyzing the Cost-Effectiveness of Code Execution in LLM-Based Program Repair](https://arxiv.org/abs/2606.26978) — Lin, Zhu, Zhou, Wang, Sun, Yang, Lo, et al., arXiv:2606.26978, 2026

**Authors.** Z. Lin, J. Zhu, M. Zhou, X. Wang, Z. Sun, R. Yang, D. Lo, et al. David Lo (SMU) is the followed name; the byline is his SMU-anchored software analytics group.

**Venue.** arXiv:2606.26978 (cs.SE), submitted June 2026. Natural targets: FSE, ICSE, ASE 2026/27.

**License.** arXiv non-exclusive — no figure embedding.

**Problem.** LLM-based agents for program repair (SWE-agent, Claude Code, Codex, OpenCode) have converged on a *generate-run-revise* loop: propose a patch, run the test suite, use the failure feedback to revise. Execution is expensive — compute, sandbox provisioning, and wall-clock latency all scale with test-run count — but its **incremental contribution** to resolve-rate has never been measured against a well-controlled counterfactual. Practitioners are running 5–20 tests per task on production infrastructure without knowing whether that number could safely drop.

**Approach.** A two-stage empirical study:

- *Descriptive stage.* Analyse **7,745 agent traces** scraped from SWE-bench leaderboard submissions. Extract per-task test-run counts, timing within the trajectory, and outcome.
- *Controlled stage.* Run **3,000 end-to-end repair attempts** across **200 SWE-bench instances** and **three agents (Claude Code, Codex, open-source OpenCode)** under **four execution paradigms** — Prohibited (no execution), Restricted, Unrestricted, and a fourth variant the abstract does not name. Compare resolve-rate and cost per paradigm.

**Headline numbers (from the abstract).**

- Code execution is used **across all agents and models**, averaging **8.8 test runs per task**; per-task frequency ranges **2 to 19**. Late-stage executions **consistently achieve higher success rates** than early-stage ones.
- On SOTA commercial agents, the resolve-rate gap between **Prohibited** and **Unrestricted** is only **1.25 percentage point**. The abstract cuts off before revealing whether the same holds for open-source OpenCode.

**Why Keep.** The 1.25 pp finding is a genuine falsification of a widely held prior: execution is the "obvious" ingredient that makes agentic repair work, and the community treats unrestricted sandbox access as table-stakes. If the finding holds up to scrutiny, execution's true value is smaller than assumed and the industry can save significant sandbox / CI cost. For the reader's own agentic tooling, the late-stage-executions-win observation is directly actionable — schedule executions toward the end of a trajectory, not the start.

**Open questions.** (i) The abstract does not disclose the fourth execution paradigm; (ii) 200 SWE-bench instances is a small sample — does the effect replicate on SWE-bench-Verified or the full 2,294-task set? (iii) The "SOTA commercial" carve-out on the 1.25 pp finding is suspicious — the gap is likely larger for weaker agents, which is what the abstract's mention of OpenCode is probably hinting at.

### 3.1 [Acoda: Adversarial Code Obfuscation for Defending against LLM-based Analysis](https://arxiv.org/abs/2606.11755) — Rao, Dong, Zhao, Li, Wang, arXiv:2606.11755, 2026

**Authors.** H. Rao, Z. Dong, Y. Zhao, H. Li, H. Wang. Not on the followed-researcher list; came through the Recommended articles alert. Affiliations not visible from the abstract snippet.

**Venue.** arXiv:2606.11755, 2026. No conference target announced.

**License.** arXiv non-exclusive — no figure embedding.

**Problem.** LLMs are being deployed to analyse third-party code for understanding, debugging, and vulnerability detection. That's a productivity win for defenders, but it's also an offensive capability against IP-protected code: an attacker can now reverse-engineer proprietary logic by pushing the source through a strong general-purpose LLM. Existing code-obfuscation techniques (control-flow flattening, identifier renaming, virtualisation) were designed against *static analysers*, not against LLMs whose semantic reasoning can see through many syntactic transformations.

**Approach.** Acoda is a **genetic-algorithm-driven adversarial obfuscator** targeting the LLM as the analyser. It:

1. Defines **8 semantics-preserving obfuscation methods** exploiting two LLM mechanisms — **safety alignment** (obfuscations that look adversarial trigger refusal) and **token-based information processing** (obfuscations that split semantic units across BPE token boundaries).
2. **Iteratively optimises** an obfuscation strategy per input via GA; fitness is measured by an **auxiliary LLM + four evaluation metrics** on how well the target LLM analyses the obfuscated code.

**Headline numbers.** Up to **70% attack success rate** (target LLM refuses or misinterprets) across **7 SOTA LLMs** — GPT-4o, DeepSeek, Qwen, Llama, Gemma. Strong cross-model transferability. Minimal runtime overhead. Semantics claimed to be preserved (the abstract cuts off before the details of the preservation check).

**Why Borderline.** The paper is interesting as a *defence* paper for IP-protection settings but the framing is uncomfortably close to *offensive* obfuscation-as-a-service — the same technique that hides proprietary code from a helpful LLM also hides malware from a security-analysis LLM. Any downstream adoption should be paired with a security-impact analysis. Methodologically the fitness-function design (auxiliary-LLM judge + four metrics) is reusable in adjacent LLM-adversarial settings, which is the main reason to Keep it in the reading queue rather than skip.

**Open questions.** (i) Do the obfuscations survive automatic normalisation (formatting, renaming to canonical identifiers, LLM-guided deobfuscation)? (ii) Is the 70% ASR measured on natural analysis prompts, or on prompts specifically shaped to trigger refusal? (iii) How does the "safety alignment abuse" component interact with the target model's refusal rate on the *unobfuscated* baseline — some fraction of the 70% may be due to the base refusal rate, not obfuscation.

### 3.2 [ConcoLixir: Reactive LLM Discovery Oracles for Python Concolic Testing](https://arxiv.org/abs/2606.26545) — Chen, Hong, Yu, arXiv:2606.26545, 2026

**Authors.** D. Chen, C. D. Hong, F. Yu. Not on followed-researcher list; Recommended articles alert.

**Venue.** arXiv:2606.26545 (cs.SE), 2026.

**License.** arXiv non-exclusive — no figure embedding.

**Problem.** Concolic testing on Python is chronically limited by three failure modes: (i) *symbolic downgrade* — library calls consume a symbolic variable and return a concrete value, losing the constraint; (ii) *hard semantic operations* — regexes, checksums, parsers, encoders resist SMT modelling; (iii) *exploration plateau* — the search returns to already-covered paths and stalls.

**Approach — LLM as bounded discovery oracle.** The paper deliberately positions the LLM as a **discovery oracle**, not a replacement for the solver and not a correctness oracle. Three roles:

1. Generate initial **seed inputs** so the concolic search does not start from `None`.
2. When the **solver fails** on a hard constraint, propose a concrete input that satisfies it heuristically.
3. When **coverage stalls**, target **uncovered code** with LLM-generated candidates.

Every LLM candidate is then executed concolically — observed coverage and collected path constraints are the only signals that guide later exploration. The LLM never asserts a specification about the program's behaviour; it only produces inputs.

**Headline numbers.** Across synthetic, real-world, and library-heavy targets, ConcoLixir improves **mean line coverage** over the no-LLM baseline by **+8.6, +15.1, and +17.0 percentage points** respectively. The gains are strongest near **semantic barriers** (checksums, regexes) and **library boundaries** (where symbolic values downgrade). Total API spend for the full evaluation is **$1.63**.

**Why Borderline (rather than Keep).** The methodological framing — LLM as bounded discovery oracle rather than solver replacement — is exactly the pattern several other 2026 papers in this reader's queue have converged on (Revelio's sanitizer-checked hypotheses, Reboot's validate-and-refine, and now this). The trend deserves to be tracked even when individual papers are workshop-scale. The two reservations that keep it out of Keep are (i) Python-only scope, and (ii) the coverage-only success metric — the paper does not (per abstract) report new bugs found, which is the harder ask for a testing tool.

**Open questions.** (i) Which LLM was used, and how does the ranking change with a cheaper model? (ii) What baseline concolic tester — CrossHair, PyExZ3, Klee-python-front-end? (iii) Does the "reactive" scheduling (only invoke the LLM on solver failure or coverage stall) generalise to fuzzers, or is it specific to path-constrained testing?

## Cross-Paper Synthesis

Two structural threads run through today's non-empty batch.

**Thread 1 — deterministic acceptance criteria for LLM outputs.** Three of the six deep-read papers — Reboot (1.1), ConcoLixir (3.2), and, if we allow the extension, Conformal Steering (2.2) — share the same architectural instinct: put a *non-LLM validator* in the loop and let the LLM propose against it. Reboot uses the Rust compiler plus test suite plus memory sanitiser as the acceptance oracle. ConcoLixir uses the concolic engine — observed coverage and collected path constraints — as the sole signal that guides continued exploration. Conformal steering uses the calibrated conformal accept-reject step to bound the target property. Compared with the 2026-06-30 Revelio report (sanitizer-checked Proof-of-Vulnerability) and 2026-06-29's THEMIS (grey-box fuzzer as oracle for WordPress plugins), a five-paper streak of "deterministic-oracle-in-the-loop" has now emerged in this reader's window. The corollary for own tooling: any agentic loop worth building this quarter should have a clearly identifiable non-LLM acceptance step, and the report where that step is missing should default to lower priority.

**Thread 2 — empirical falsification of default assumptions in LLM-agent stacks.** The David Lo group's "To Run or Not to Run" (2.3) reports a **1.25 pp** resolve-rate gap between Prohibited and Unrestricted execution on SOTA commercial agents — a plausible falsification of the widely held prior that unrestricted sandbox execution is table-stakes for agentic APR. This is a rare class of paper (empirical study designed to falsify a well-motivated assumption in a hot subfield) and is worth cross-referencing against the earlier trace-analysis papers of 2024–2025 (Yang et al.'s SWE-agent scaling laws, Xia et al.'s agentic-repair post-mortems). If the 1.25 pp holds up, the cost-optimal APR agent architecture may look very different from today's.

The two threads intersect in a useful way. If deterministic-oracle-in-the-loop is the architectural pattern that lets LLM outputs be trusted, then the specific *cost* of the oracle step (test execution, compilation, sanitiser run) becomes the top variable to tune — which is exactly what "To Run or Not to Run" is measuring. Reboot's milestone-based translation is an implicit answer to the same question in a different domain: run the sanitiser at every milestone (small, cheap oracle calls), not once at the end.
