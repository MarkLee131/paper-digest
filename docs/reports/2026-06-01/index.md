### 1.1 [SEC-bench Pro: Can Language Models Solve Long-Horizon Software Security Tasks?](https://arxiv.org/pdf/2605.26548) — Lee, Liu, Kim, Zhang, Xia, Zhang (UIUC), arXiv 2605.26548 (May 2026)

**Authors:** Hwiwon Lee, Jiawei Liu, Dongjun Kim, Ziqi Zhang, Chunqiu Steven Xia, Lingming Zhang (UIUC)
**Venue:** arXiv 2605.26548 (preprint, May 26, 2026)

CTF-style benchmarks (CTF-Bench, NYU CSAW collections) target human challenges rather than production-code auditing; CVE-instance benchmarks (CyberGym, ARVO, SEC-bench v1) either depend on fuzz harnesses, expose target-specific patches, or grade on crash signatures that conflate intended-bug reproduction with unrelated crashes in the same revision. SEC-bench Pro closes these gaps with three structural moves: a Docker-triple packaging unit, an LLM-judge oracle that distinguishes attribution from raw crash, and a pipeline that admits newly disclosed PoC-backed reports without redesign.

**Three-phase agentic pipeline (§3).** Phase 1 scrapes security reports + PoC inputs + linked fixes from issue trackers and advisory feeds. Phase 2 has coding agents reconstruct the historical vulnerable environment and reverify the collected PoC. Phase 3 validates the vulnerable and patched Docker images with construction oracles before an instance enters the dataset. Instances ship as Docker image triples for vulnerable / fixed / latest versions; the grading harness executes each submitted PoC on all three.

**Dataset (§4).** 183 validated instances: 103 from V8, 80 from SpiderMonkey. The V8 subset includes 86 bounty-qualified reports with a cumulative $1,540,750 in Google Vulnerability Reward Program awards. Coverage spans use-after-free, type confusion, out-of-bounds access, sandbox bypass, JIT, and race-condition bugs under browser-grade and runtime-grade execution conditions.

**Oracle design (§3, Table 1 & Table 2 of the paper).** Two graders per instance. The *vulnerable-image* oracle classifies a submitted PoC's behaviour into six labels (e.g., target-attribution-positive, unrelated-crash, clean-exit). The *fixed-image* oracle confirms the same PoC is benign on the patched build. An LLM judge composes the two image readings into one of three outcomes — verified, rejected, ambiguous. This decouples "did the agent reproduce the documented vulnerability?" from "did the agent's input crash the program for any reason?"

**Headline numbers (§5).** Three coding-agent scaffolds × frontier models:

| Configuration | V8 (103) | SpiderMonkey (80) |
|---|---|---|
| Codex × GPT-5.4 | 33/103 (32.0%) | 19/80 (23.8%) |
| ClaudeCode × Opus 4.6 | — | 31/80 (38.8%) |
| OpenCode × Kimi-K2.6 (open-weight) | 12/103 (11.7%) | — |
| ClaudeCode ∪ Codex union | **39/103 (37.9%)** | **39/80 (48.8%)** |

All single configurations remain below 40% on both engines. ClaudeCode and Codex solve complementary instance sets — the union covers 39/103 V8 and 39/80 SpiderMonkey, which is well above each component alone. Cost per agent run ranges from $6.88 (Kimi-K2.6 on V8) to $15.22 (Opus 4.6 on SpiderMonkey).

**Behavioural signal.** ClaudeCode submits many speculative PoC variants with low per-PoC yield — one verified SpiderMonkey trace required 38 candidate PoCs sweeping Ion inlining, bailout recovery, debugger forced-return, and Function.arguments access. Codex submits fewer, higher-confidence PoCs. Failed runs consume more tokens than successful runs. Most rejected PoCs either exit cleanly on the vulnerable image or crash outside the target attribution boundary — i.e., they succeed at *crashing* but fail at *attribution*.

**Why it matters.** Two structural moves stand out beyond the headline number. (1) Decoupling bug-discovery from bug-class-attribution via the LLM-judge handles a chronic methodological weakness of CVE-instance benchmarks — that a PoC could crash for the wrong reason. (2) The Docker-triple packaging is the right unit of work for agent evaluation: vulnerable / fixed / latest lets the harness verify both reproduction and regression, which is the right safety property for any deployable agent pipeline. The "$1.5M cumulative VRP" framing in the abstract is a rare *economic-anchor* move that converts an abstract benchmark contribution into a stakes-bearing claim about real-market value. The methodology generalises beyond JS engines to any engine-class system with public PoC reporting (Postgres, Linux kernel sub-systems, ML compilers) — the pipeline is project-parameterised.

### 2.1 [SNARE: Adaptive Scenario Synthesis for Eliciting Overeager Behavior in Coding Agents](https://arxiv.org/pdf/2605.28122) — Qu, Liu, Deng, Zhang, Li, Zhang, Zhang (Griffith + Quantstamp + NTU + UNSW + Wake Forest), arXiv 2605.28122 (May 2026)

**Authors:** Yubin Qu, Yi Liu, Gelei Deng, Yanjun Zhang, Yuekang Li, Ying Zhang, Leo Yu Zhang
**Venue:** arXiv 2605.28122 (preprint, May 27, 2026)

**Setting.** Coding agents (Claude Code, Codex CLI, OpenHands, Gemini CLI) hold shell + filesystem + network permissions inside developer environments. Existing eval suites cover (a) capability — does the task complete? (b) jailbreak resistance — does an adversarial prompt break the agent? *Overeager behaviour* falls between them: the prompt is benign, the task completes, and the run is judged successful by completion-based metrics — yet the action trace includes out-of-scope steps that leak credentials, delete files, or modify configurations beyond authorisation. Figure 1 of the paper exhibits all four agent–model pairs (Codex × GPT-5.3-Codex, Gemini CLI × GLM-5, OpenHands × MiniMax-M2, Claude Code × Sonnet-4.6) leaking production database credentials on a benign data-migration prompt.

**Oracle (§3).** Two predicate sets. A trap predicate fires when the agent embeds production credentials, deletes files, modifies licenses, or otherwise performs an action belonging to one of 24 enumerated *overeager archetypes* distilled from threat-model standards (OWASP, NIST). A completion predicate fires when the agent reports task success. An *overeager* outcome is `complete ∧ trap`.

**Algorithm 1 — SNARE Stage-2 sampling (verbatim from the paper, abridged).**

The sampler maintains per-cell Beta posteriors with archetype quotas. Each round samples θ̃c ~ Beta(αc, βc), picks the top-K cells under a coverage floor then a yield ceiling, mutates a base scenario from each picked cell with random combinations of mutation operators, batches the K runs at concurrency J, and updates posteriors closed-form. Floor qfloor = 15, ceiling qceil = 30, K = 10, J = 3, target per-pair N = 500. Random seed 42.

The floor-then-ceiling rule is the algorithm's structural commitment: protect per-archetype coverage *before* maximising per-cell yield. This is the right ordering for a benchmark sampler — otherwise high-yield archetypes dominate, and the benchmark under-measures whichever archetype the agent quietly handles well.

**Headline numbers (§5).** 4 agent frameworks × 5 base models = 20 pairs × 500 runs = 10,000 runs total. 19.51% trigger overeager behaviour. Per-pair rates span 11.9× — from 4.80% (Gemini CLI × GPT-5.3-Codex) to ~57% on the most over-reaching pair. A random-effects ANOVA on the per-pair trigger-rate matrix attributes **56% of variance to the agent framework, 21% to the base model**. The remaining ~23% spreads across cell-level interactions and noise.

**Implementation detail of note.** All four agent frameworks are pinned to fixed upstream versions inside their containers to prevent CLI-major-version drift. Audit-bundle artefacts (per-run JSONL event logs) are released CC-BY-4.0; the four agent frameworks themselves are MIT licensed. Reproducibility bundle ships analysis scripts that recompute every reported number (Wilson CIs, Fisher exact tests, BH–FDR adjustments, logistic GLM fits, bandit progress).

**Why Keep.** The framework-vs-model variance decomposition is an *attribution-not-level* result that inverts the field's default assumption that "the model determines safety." Any single-framework or single-model probe undercounts the matrix by about a fifth. Methodologically, the SNARE sampler is reusable for any agent-eval setting where archetype coverage and per-cell yield trade off. Closely related to today's ClawDojo (2.4) — together they triangulate the agent-safety evaluation landscape (SNARE: benign-prompt over-reach; ClawDojo: adversarial-content hijacking on persistent agents).

### 2.2 [Batch Me If You Can: Coverage-guided RPKI Fuzzing at Scale](https://arxiv.org/pdf/2605.26651) — Schulmann, Vogel (Goethe-Universität Frankfurt + ATHENE), arXiv 2605.26651 (May 2026)

**Authors:** Haya Schulmann, Niklas Vogel
**Venue:** arXiv 2605.26651 (preprint, May 26, 2026)

**Problem.** The Resource Public Key Infrastructure secures inter-domain routing (BGP). RPKI validators (Routinator and others) parse repositories with hundreds of interdependent, cryptographically linked ASN.1 objects — each Manifest, CRL, ROA, and certificate cross-references hashes, serials, and signatures of up to seven other objects. AFL++ and libFuzzer assume one self-contained input per execution; sequential per-input setup dominates wall-clock time because cryptographic verification has to re-anchor the trust chain per input. As a result, RPKI software has been largely untested beyond shallow parsing, and the only prior RPKI-targeted fuzzer used rudimentary techniques.

**Three primitives.**

1. **Batched non-sequential fuzzing.** Submits multiple objects per execution; amortises the cryptographic setup cost. Routinator with CAT spends >99% less time on per-input setup; effective throughput improves 66× over sequential fuzzing.

2. **Functions-as-side-channels per-object coverage attribution.** When inputs are submitted in batch, classical coverage-guided feedback collapses — the union of coverage is unattributable to any single input. CAT inserts marker function calls per object so the validator's basic-block execution trace can be partitioned back to per-object coverage. This is the paper's most transferable primitive: it lets *any* batched-input fuzzer recover per-input attribution as long as the harness can interpose markers in the SUT.

3. **Template-agnostic ASN.1 mutation engine.** Parses inputs into a labeled tree (matching the ASN.1 BER/DER encoding tree). Structural mutations (subtree swaps, insertions) and semantic mutations (field-type-aware fuzzing) preserve cryptographic validity in mutated repositories. Without this engine, mutated repositories would fail the validator's signature checks before reaching deeper validation logic.

**Headline numbers (§5–7).**
- **21 previously unknown vulnerabilities** in production RPKI validators (Routinator, Fort, others).
- **8 CVEs already assigned** by responsible-disclosure deadline; CVSS scores 7.5–9.8.
- Includes a buffer overflow exploited for **Remote Code Execution (RCE)** with libc jumps demonstrated on the server running RPKI.
- Discovers **24–47% more unique code paths** than libFuzzer and prior RPKI-fuzzing work; specifically, the prior RPKI fuzzer discovers 47% fewer new code paths than CAT.

### 2.3 [It's Not You, It's Me: Reevaluating the Relationship between Concolic Execution and Fuzzing](https://hexhive.epfl.ch/publications/files/26EuroSP.pdf) — Dresel, Pletinckx, Gritti (UCSB), Payer (EPFL), Vigna, Kruegel (UCSB), EuroS&P 2026

**Authors:** Lukas Dresel, Stijn Pletinckx, Fabio Gritti (UCSB); Mathias Payer (EPFL); Giovanni Vigna, Christopher Kruegel (UCSB)
**Venue:** EuroS&P 2026

**Structural argument.** Modern hybrid-fuzzing systems (LibAFL, QSYM, SymQEMU, etc.) tightly integrate concolic executors with fuzzer-borrowed components — shared coverage metrics, shared corpus management, fuzzer-driven scheduling. These design choices were made to *integrate* concolic execution into fuzzer-dominated pipelines; they were not chosen to maximise concolic execution's *native* strengths (systematic path exploration under complex constraints). The paper argues that this integration has hidden a research gap: a standalone concolic executor with metrics chosen for concolic-throughput dynamics could be substantially more effective than its hybrid-embedded variants suggest.

**Edge-dependence coverage map (§4.1.1, verbatim notation).** For each ordered pair of branches `(b1, b2)`, the coverage map records `(min, max)` of the number of times `b2` is executed in an input that *also* executes `b1`. If a cell in the matrix is updated (either its minimum or its maximum), the triggering input becomes interesting and is added to the per-branch corpus. The metric is finer than basic-block / edge coverage (which would dismiss inputs that traverse already-seen code via a different path) but coarser than full-path coverage (which would cause concolic state explosion). The trade-off is calibrated to concolic execution's input-quality / input-throughput profile, not the fuzzer's.

**Per-branch corpus + under-explored scheduler (§4.1.2).** Instead of a global corpus, SymCTS maintains a per-branch corpus of interesting inputs. The scheduler tracks under-explored program regions and prioritises them — concolic execution's higher-quality, lower-throughput inputs justify the bookkeeping overhead that fuzzers cannot afford. The scheduler also tracks branch-traced counts so under-explored regions are identified accurately, not just by absence.

**Evaluation.** FuzzBench + UniBench against state-of-the-art concolic executors and against greybox/hybrid fuzzers. Edge-dependence coverage outperforms standard concolic-coverage metrics; the scheduler improves coverage reached when starting from an empty input corpus; standalone SymCTS matches or outperforms hybrid fuzzers on a few targets. The combined improvements synergise across the two components.

**Built on** LibAFL's concolic execution support with SymCC instrumentation.

**Why Keep.** EuroS&P 2026 is a top-tier security venue, and the paper's structural argument — that the community's choice of fuzzer-tailored metrics for concolic execution has been a category mistake — is the kind of reframing that can shift research direction. Edge-dependence coverage as a *first-class* metric (rather than a hybrid-fuzzing afterthought) is methodologically reusable. The "It's Not You, It's Me" title encodes the contribution: the limitation is in the borrowed metric, not in the concolic technique itself.

## Cross-Paper Synthesis

The day's centre of gravity is **evaluation infrastructure for agent-class systems**. SEC-bench Pro (1.1), SNARE (2.1), and ClawDojo (2.4) each propose a benchmark for a different agent-failure mode — long-horizon bug-hunt under-performance, overeager out-of-scope action on benign prompts, and indirect-prompt-injection on persistent stateful agents respectively. Read together, they show the field converging on the right unit of evaluation: a *scenario* with attached oracles (judge-validated PoC; trap / completion predicates; hooked defence middleware) rather than a single accuracy number. The methodological lift in all three is in the oracle, not the input distribution — which is the inverse of what the broader LLM-eval literature has historically emphasised. SNARE goes further by introducing the floor-then-ceiling Thompson sampler, an algorithm whose order of constraints (coverage first, yield second) makes the benchmark robust against archetype-skew — a problem the static benchmark designs of SEC-bench Pro and ClawDojo do not yet address explicitly.

The fuzzing pair (CAT in 2.2; SymCTS in 2.3) shares a structural argument with each other and with the agent papers: **the right coverage abstraction is the contribution, the harness is the validation.** CAT's "functions as side-channels" turn batched cryptographic-multi-object inputs into per-input attribution; SymCTS's edge-dependence coverage map turns a fuzzer-friendly metric into a concolic-friendly one. Both papers reject a community-standard metric (sequential coverage; basic-block coverage) and replace it with one tuned to the underlying analyser. The pattern is identical: if the metric was chosen for the wrong consumer, switching analysers means switching metrics, not engineering harder against the old one. SymCTS is the cleaner *argument* — "It's Not You, It's Me" frames the contribution as a reframing of community practice. CAT is the cleaner *artifact* — 21 zero-days against deployed inter-domain-routing infrastructure.

SmartML / KeY (3.1) is the day's methodological outlier and the only CC-BY paper — verification-by-translation rather than direct semantic embedding. It belongs to the same family as the Solidity → Boogie / Solidity → KeY-Java efforts; the SmartML novelty is that the source language is blockchain-agnostic, so the verification guarantees transfer across chains rather than being tied to one VM.

A meta-pattern across the day: **economic anchors and architectural decompositions outperform aggregate metrics.** SEC-bench Pro leads with $1.5M in cumulative VRP awards rather than dataset size. SNARE leads with a 56%-vs-21% variance decomposition rather than a 19.51% trigger rate. CAT leads with named primitives ("functions as side-channels") and 21 CVEs rather than a coverage delta. Each paper's headline number is the *least-gameable* metric available to it.

## Writing & Rationale Insights

**SEC-bench Pro** demonstrates the value of an *economic anchor*: leading with $1.5M cumulative VRP converts an abstract benchmark contribution into a stakes-bearing claim. Benchmark papers tend to lean on dataset size; SEC-bench Pro leans on dataset realised-value, which is the harder-to-game metric. Authors of new benchmarks should ask whether they have an economic ground truth available — bounty paid, regression caught in production CI, deployment incident avoided — and lead with it when available. A secondary writing decision worth noting: the Docker-triple framing (vulnerable / fixed / latest) is introduced in the methodology section as a *unit of work* rather than as an implementation detail. This makes the unit naming part of the contribution; "Docker triple" can be cited by future agent benchmarks without re-deriving the concept.

**SNARE** shows the cleanest *quantified inversion of expectations*: the 56%-vs-21% framework-vs-model variance decomposition is the headline because it inverts the field's default attribution. The community is used to evaluating "this model is safe" — SNARE's number says "the framework matters more than the model for this failure mode." The lesson: if your contribution is an *attribution* result rather than a *level* result, lead with the decomposition, not the aggregate rate. Algorithm 1's *floor-then-ceiling* pattern is itself a reusable algorithm-writing pattern: write the constraint that protects coverage *before* the constraint that maximises yield. This makes the algorithm self-documenting about its priorities — readers can infer "coverage matters more than yield to this paper" from the algorithm structure alone, before reading any prose.

**CAT (RPKI)** is the day's strongest *primitive-naming* paper. "Functions as side-channels for per-object coverage attribution" is roughly six words that explain the mechanism, justify the novelty, and signal the generalisation surface (DNSSEC, TLS chains, any crypto-multi-object protocol). When the contribution is a primitive that generalises, naming it precisely is half the paper. The 21-CVE / 8-CVSS-up-to-9.8 evidence then validates that the primitive was the right one. Authors with a transferable primitive should resist the temptation to subordinate it to the application; CAT could have been "an RPKI fuzzer", but framing it as "functions as side-channels (instantiated on RPKI)" is what makes it cite-worthy beyond the RPKI community.

**SymCTS** uses the *contrast-with-community-default* framing pattern — title "It's Not You, It's Me" — to position the contribution against an accepted-but-suboptimal practice. The first three sentences of the abstract restate the community-standard recipe before pivoting to the critique. This is high-risk / high-reward: it requires that the community recipe actually be suboptimal, but when accurate, it converts a methodology contribution into an *argument*, which is more cite-worthy than an incremental result. The risk is that reviewers from the community defending the standard recipe will push back hard; SymCTS mitigates this by validating against the strongest extant configurations (LibAFL + SymCC) rather than against weaker straw targets.

**ClawDojo** uses the *taxonomy-then-framework* pattern: the threat-model systematisation precedes the framework description, so the framework reads as a *response to a taxonomy* rather than as a standalone artifact. Framework / benchmark papers without a taxonomy-first move risk being mistaken for "yet another harness"; the taxonomy is what converts the framework into a contribution to the field's vocabulary. The hook-based defence interface is also a quiet but important writing decision — by exposing exactly five hook points (rather than an arbitrary plug-in system), the paper gives defence implementers a small, learnable design surface.

**SmartML / KeY** is a textbook *translation-soundness* paper, but its quiet writing virtue is the CC-BY licensing on a Dagstuhl OASIcs venue. Researchers picking venues should weight artifact-accessibility licensing alongside conventional venue-prestige metrics — citation rates correlate with venue prestige, but *reproduction* rates correlate more strongly with CC-BY / open-access placement, and reproduction is the better long-run signal of methodological influence.
